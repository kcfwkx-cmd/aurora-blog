搭博客搭了一阵子，今天趁着等 Claude Code 改 CSS 的空隙，把这套东西的逻辑捋一捋。

## 两个地盘，一个原则

整个项目分两块：**你的地盘**和**主题的地盘**。

你的地盘是根目录，包括 `_config.yml`（Hexo核心配置）、`_config.solitude.yml`（主题功能开关）、还有 `source/` 文件夹——你写的所有文章、页面、数据文件都在这里。这块地方是安全的，怎么改都不会被覆盖。

主题的地盘是 `themes/solitude/`，是别人写的代码。里面有模板、样式、脚本。原则上不要动，因为主题一升级就会被覆盖。但有时候配置解决不了的问题，只能进去改——改了就要记录下来。

**一个原则：能用配置解决的，绝不动主题文件。**

## 三种文件语言

接触 Hexo 主题会遇到三种陌生格式：

**Pug** 是模板语言，本质是 HTML 的简写版。不用写开闭标签，用缩进表示层级。`div.nav-item` 就是 `<div class="nav-item">`。主题的页面结构全用它写。

**Stylus** 是 CSS 的增强版，支持变量和嵌套，最终编译成普通 CSS。控制颜色、字体、间距、布局。你在浏览器里看到的视觉效果，都是它决定的。

**YAML** 是配置语言，`_config.solitude.yml` 用的就是这个格式。用缩进和冒号表示层级，专门用来写配置，人读起来最直观。

## 一个请求的旅程

你在浏览器里打开博客首页，背后发生了什么？

Hexo 找到 `layout/index.pug`，这是首页的模板。模板里引用了导航栏、侧边栏、文章列表等组件，这些组件都在 `layout/includes/` 里。渲染过程中读取 `_config.solitude.yml` 决定显示什么内容，加载 `source/css/` 决定怎么显示，加载 `source/js/` 处理交互。最后你看到的页面，是这些东西组装出来的。

## `public/` 是什么

每次执行 `hexo g`，Hexo 就把上面所有东西编译一遍，生成纯静态的 HTML/CSS/JS 文件，放到 `public/` 文件夹里。部署到服务器上的就是这个文件夹。

所以博客的本质是：**你写 Markdown，Hexo 编译成静态网页，服务器托管这些网页。**

没有数据库，没有后端，访问速度极快，也极难被攻击。

但够用了。知道出了问题去哪里找，知道改了什么要记录，知道边界在哪里——这就够了。
