# Cursor 协作规范（Aurora 博客项目）

> 用途：约定我们在这个项目里如何协作、如何提需求、如何记录改动，避免“改了什么/为什么改/怎么回滚”说不清。

## 开工约定（每次开始前）

- **触发词约定**
  - 你发 **“开工”**：我会先读 `cursor.md` 与 `Aurora_progress.md`，再开始处理新任务
  - 你发 **“收工”**：我会把当天改动/结论补记到 `Aurora_progress.md`（必要时再补到 `aurora.md`）

## 修改优先级（从高到低）

1. **改配置**：优先改博客根目录 `_config.yml` / `_config.aurora.yml`
2. **用站点自定义资源覆盖**：优先在 `source/` 下添加资源（如 `source/css/custom.css`）
3. **注入自定义 CSS/JS**：优先使用 `_config.aurora.yml` 的 `injects` 注入
4. **最后才改主题包**：尽量不改 `node_modules/hexo-theme-aurora`（升级会覆盖）
   - 如果不得不改：必须同步记录到 `aurora.md` 的“主题内置文件修改记录”

## 终端与命令执行约定

- **终端命令由你执行**（例如 `hexo clean; hexo g; hexo s`、端口排查等）
- 我提供：
  - 需要跑的命令（PowerShell 兼容写法）
  - 改完配置/文件后提醒你运行哪条命令

## 需求描述建议（你提需求时）

- **给目标**：你想要“长什么样/达到什么效果”
- **给范围**：只改配置？还是允许加 `source/css/custom.css`？能不能动主题包？
- **给截图**：UI/样式类问题尽量截图（你已经在用这个方法，效率很高）

## 记录与回溯

- **凡是涉及 Aurora 主题行为改变**：在 `Aurora_progress.md` 写一条当天记录
- **凡是可能被主题升级覆盖的改动**：在 `aurora.md` 写一条可迁移的记录

## 我更好的建议（可选）

- 如果你希望“每次开工我都强制先读这些文件”，可以加一条 Cursor 规则（`.cursor/rules/`）。  
  这样能更稳定地“带着记忆开工”。你想要的话我可以帮你把规则文件也生成好。
